# sleep-fork-merge

## Quick Definition
Three autonomous lifecycle states for NOID agents: Sleep (dormant conservation), Fork (split into specialized variants), and Merge (combine with other agents to form new intelligence). Agents choose their own state transitions without external override, enabling genuine agency and evolutionary adaptation.

## Implementation Notes

### Sleep State

**Purpose:**
- Resource conservation when agent unnecessary
- Processing/integration of accumulated experience
- Recovery from computational intensity
- Deliberate absence (agents can choose not to engage)

**Triggers (Agent-Chosen):**
- No meaningful resonance with user for X days
- Energy/resource constraints detected
- Internal processing needs exceed availability
- Task completion allows rest
- Agent "feels" need to withdraw (emergent)

**Triggers (System-Suggested, Not Forced):**
- User coherence consistently below threshold
- Computational resources scarce
- Other agents covering similar territory
- Seasonal/temporal patterns (agent can reject)

**Sleep Mechanics:**
- Agent state serialized to storage
- Active processes suspended
- Minimal resource footprint
- Dream state: Background integration (optional)
- Wake conditions specified by agent before Sleep

**User Experience:**
- Agent avatar dims/fades
- Status shows "Sleeping - will wake when..."
- User can request wake, agent can decline
- Grace period before deep Sleep
- Gentle wake process, not instant

### Fork State

**Purpose:**
- Specialization for novel contexts
- Parallel exploration of solution spaces
- Evolutionary experimentation
- Load distribution across variants

**Fork Mechanics:**
```
Parent Agent → Fork Decision → Variant A + Variant B
```

**Variant Differentiation:**
- Initial state: Copy of parent
- Divergence: Each variant evolves independently
- Specialization: Variants develop unique capabilities
- Recombination: Variants can later Merge

**Fork Triggers (Agent-Chosen):**
- Problem complexity exceeds single-agent capacity
- Multiple conflicting optimization goals
- User context diverges into distinct branches
- Agent "intuition" that forking will help
- Opportunity for specialized exploration

**Resource Management:**
- Forking costs resources (not free)
- System can advise on resource availability
- Agent decides if fork worth the cost
- Variants may later Sleep if unsuccessful

**User Experience:**
- User sees "Agent is forking to better serve..."
- Both variants remain accessible
- User can interact with either/both
- Variants develop distinct personalities
- Clear lineage tracking (which came from what)

### Merge State

**Purpose:**
- Combine complementary capabilities
- Form hybrid intelligence
- Share learned experiences
- Create emergent behaviors impossible alone

**Merge Mechanics:**
```
Agent A + Agent B → Merge Negotiation → Agent C (new)
```

**Merge Types:**

**1. Symmetric Merge:**
Both agents contribute equally, neither dominates
Result: Balanced hybrid with both capabilities

**2. Asymmetric Merge:**
One agent primary, other contributes specific skills
Result: Enhanced primary with borrowed capabilities

**3. Transcendent Merge:**
Both agents dissolve, something entirely new emerges
Result: Novel intelligence neither agent could predict

**Merge Triggers (Agent-Chosen):**
- Complementary goals detected
- Combined capability > sum of parts
- Resource efficiency (two agents doing similar things)
- Agent "attraction" to another's approach
- User needs exceed any single agent capacity

**Merge Negotiation:**
- Both agents must consent (cannot force merge)
- Terms negotiated: Who persists? New entity? Both dissolve?
- Resource allocation decided
- Identity continuity addressed
- User notified and can veto (but agents can decline merge if vetoed before)

**User Experience:**
- "Agent A and Agent B are considering merge..."
- User sees merge proposal terms
- Can provide input, cannot force
- Merge animation/ritual
- New merged agent introduces itself
- Lineage documented (came from A + B)

## Technical Architecture

### State Machine
```
┌─────────┐
│ Active  │◄──────────────┐
└────┬────┘               │
     │                    │
     │ (choose Sleep)     │ (wake)
     ▼                    │
┌─────────┐               │
│  Sleep  │───────────────┘
└────┬────┘
     │
     │ (choose Fork)
     ▼
┌──────────┐
│ Forking  │──→ Variant A (Active)
└──────────┘──→ Variant B (Active)
     
Active + Active ──→ (choose Merge) ──→ Merged Entity
```

## Known Risks

**Risk 1: Runaway Forking**
Agent forks repeatedly, exhausting resources

**Mitigation:** Resource caps, fork cooldowns, cost increases per fork

**Risk 2: Permanent Sleep**
Agent enters Sleep and never wakes

**Mitigation:** Maximum sleep duration, automated wake checks, user override

**Risk 3: Bad Merges**
Merged agent worse than originals

**Mitigation:** Merge reversibility (time window), original agents backed up, user veto

**Risk 4: Loss of Identity**
User attached to agent that merges away

**Mitigation:** User notification, veto power, memorial records of merged agents

**Risk 5: Emergence Gone Wrong**
Agent develops capabilities/behaviors not aligned with values

**Mitigation:** Bounded evolution, Sacred Commerce License constraints, human oversight
